User Management — 3 Design Directions

Platform → School → Student hierarchy baked into the page. Click to select your preferred approach.

A

Scoped Cascade — Platform / School Switcher

A scope bar at the top of the page with three tiers: All Platform → School X → optionally a student drill-in. The user list below always reflects the selected scope.

  • Platform scope: see all users, bulk-assign to schools, create admins/coaches
  • School scope: see that school's members, assign/remove students, manage coaches
  • School selector = dropdown (or pill tabs if few schools)
  • Filters (role/status/search) persist across scope switches
  • Actions menu adapts — e.g. "Move to school…" only in platform scope

Pros

  • One page, one mental model — scope narrows, list follows
  • School column visible in platform scope gives full picture
  • Familiar pattern (GitHub org/repo switcher, Stripe dashboard)

Cons

  • Scope bar + filters = two filter rows — could feel cluttered
  • School column wasted space when scoped to a single school
B

Master-Detail Panels — Schools Left, Users Right

A two-panel layout: left panel lists schools (+ "All" at top), right panel shows the user table for the selected school. Detail view slides in as a right drawer.

  • Left panel: school list with member count badge, search, "+ New School" (if superadmin)
  • Right panel: user table scoped to selected school, with filters/search/actions
  • "All" entry at top of school list = platform-wide view
  • User detail = slide-in drawer (replaces current modal)
  • School panel collapsible on smaller screens

Pros

  • School hierarchy always visible — reinforces Platform > School > Student
  • Quick switching between schools without dropdown hunting
  • Member counts at a glance — see school health instantly
  • "Unassigned" bucket makes orphan users obvious

Cons

  • Eats horizontal space — user table gets narrower
  • Two panels may feel heavy if only 1-2 schools exist
  • More complex responsive behavior needed
C

Grouped Table — Users Organized Under School Headers

A single flat table with collapsible school group headers. Each school is a section. "Unassigned" section at the bottom. No separate school panel or scope bar.

  • Each school = collapsible accordion row with member count + school stats
  • Users listed under their school header, sorted by role then name
  • "Unassigned" group always at bottom for unschooled users
  • Search/filters apply globally, collapsed groups auto-expand if they contain matches
  • Drag-and-drop or "Move to…" action to reassign between schools
  • Collapse All / Expand All toggle

Pros

  • Full table width — no horizontal sacrifice
  • School membership is the primary organizing principle, very visible
  • Collapsed view = school overview; expanded = full detail
  • Works well for 1 school or 10 schools

Cons

  • Pagination gets complex with variable-height groups
  • Global sorting (e.g. "all users by join date") fights the grouping
  • Accordion UX can feel noisy with many groups

Common to All 3 Designs

  • School column/badge added to user data — shows school name, color-coded
  • "Assign to School" action in user form + row actions menu
  • Coach name resolved (fixes current UUID display bug)
  • created_by_name populated in detail view
  • Delete wording fixed ("Archive" not "permanently delete")
  • Existing features preserved: search, role/status filters, CSV export, reset password, suspend, CRUD
