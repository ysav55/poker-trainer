Phase 1 — Auth Layer Consolidation

What's changing, why it matters, and what we're touching

🔑 The Problem in Plain English

Right now, the server uses a secret key to sign and verify login tokens (like a wax seal on an envelope). The code that does this verification is buried inside the database module — which is like asking your filing clerk to also be the bouncer. That creates two problems:

⚠️ Any test that touches the database also has to pretend to be the auth system. They're tangled together.
⚠️ There's no single place that "owns" the login token logic — it's split across two unrelated files.
✅ After Phase 1: one dedicated file (JwtService.js) handles all token signing and verification. Everything else just calls it.
Before vs After — How the token flows

❌ Before — Scattered Auth

🖥️ index.js — Login Handler calls jwt directly
↓ jwt.sign() lives here
🗄️ HandLoggerSupabase.js also has jwt.verify
↓ called by
🖥️ index.js — requireAuth imports from DB layer
🔴 3 different places that touch JWT. To change how auth works, you need to edit multiple files.

✅ After — Single Source of Truth

🔐 JwtService.js new — owns jwt
↓ imported by
🛡️ requireAuth.js new — middleware
↓ used by
🖥️ index.js — Login Handler uses JwtService.sign()
🟢 One file owns JWT. Everything else just calls it. Database layer has no auth knowledge.
Files Being Changed
Create
server/auth/JwtService.js
New file. The only place in the entire codebase that imports jsonwebtoken. Exposes two functions: sign(payload) to create a token, and verify(token) to validate one.
Create
server/auth/requireAuth.js
New file. Express middleware that checks if a request has a valid token. Replaces the inline version currently living in index.js. Only imports from JwtService.
Create
server/auth/requireRole.js
New file. A factory that creates role-checking middleware — e.g. requireRole('coach') blocks non-coaches from sensitive endpoints.
Modify
server/index.js
Login handler swaps jwt.sign() → JwtService.sign(). The inline requireAuth function is replaced with an import from the new module. No logic changes — just wiring.
Delete
server/db/HandLoggerSupabase.js — authenticateToken()
The authenticateToken function is removed entirely from the database layer where it never belonged. The DB module stops importing jsonwebtoken.
Modify
server/db/__tests__/REST.api.test.js
Remove the authenticateToken stub from the HandLoggerSupabase mock — it no longer exists there. Done once here, never again.
Modify
server/game/__tests__/socket.integration.test.js
Same — remove authenticateToken stub. This was the only reason these integration tests had to know about JWT at all.
Test
server/auth/__tests__/JwtService.test.js
New tests for the new module. Covers: valid tokens pass, expired tokens fail, tampered tokens fail, missing secret exits the process.
Step-by-Step Execution Order
1
Create server/auth/ directory and JwtService.js
Write sign() and verify() backed by the SESSION_SECRET env var. If the secret is missing, verify() throws — fail-fast behaviour is preserved.
2
Create requireAuth.js and requireRole.js
Pure Express middleware files. requireAuth calls JwtService.verify() and puts the decoded payload on req.user. requireRole('coach') checks req.user.role.
3
Update index.js — swap login to use JwtService.sign()
One-line change in the login handler. The requireAuth function at the top of index.js is replaced with const requireAuth = require('./auth/requireAuth').
4
Delete authenticateToken from HandLoggerSupabase.js
Remove the function body, the jwt import, and the export. The DB module is now pure data — no auth knowledge.
5
Clean up both test mocks
Remove authenticateToken: jest.fn() from REST.api.test.js and socket.integration.test.js. This is the last time we touch these mocks for auth reasons.
6
Write JwtService.test.js and run full suite
Confirm all 900+ existing tests still pass, plus the new auth unit tests.
Why This Unlocks Everything Else
🔓

Unblocks Phase 3

Splitting the database layer (Plan 03) is blocked until authenticateToken moves out — otherwise the DB module carries auth baggage into the new repos.

🧪

Cleaner Tests

Tests that mock the DB no longer need to also stub auth functions. Each mock is now smaller and easier to read.

🛡️

Easier to Audit

When something goes wrong with tokens, you look in one place: server/auth/. Not across two unrelated modules.

🔧

Future-Proof

Want to change token expiry, add refresh tokens, or switch JWT libraries? Change one file. Nothing else breaks.

🧯 Risk Assessment

This is a low-risk refactor. No game logic changes. No database schema changes. No client changes. Just moving where auth code lives.

Risk level
Very Low
Test coverage
Strong — 900+ tests catch any regression
