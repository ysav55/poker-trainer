2 · Data Flow: Hand End → Tags in DB

1
Hand ends in GameManager gameLifecycle.js calls HandLogger.endHand() — writes board, final_pot, winners to DB
2
Analysis triggered automatically .then(() => AnalyzerService.analyzeAndTagHand(handId)) — non-blocking, errors logged but don't break game
3
buildAnalyzerContext(handId) 3 parallel DB queries: hands, hand_actions, hand_players → enriched with sizingRatio, positions, byStreet map, memoized evaluateAt(playerId, street), botPlayerIds Set
4
Promise.allSettled([…10 analyzers…]) All 10 analyzers run in parallel. Each returns TagResult[]. One analyzer crashing doesn't block others.
5
Post-processing Validate TagResult shape → deduplicate hand/player-level tags → filter bot-player tags
6
replaceAutoTags(handId, tagRows) Atomic: DELETE old auto/mistake/sizing tags → INSERT new batch. Coach tags are never touched.
7
Client reads tags via REST GET /api/hands/history (filtered list) · GET /api/hands/:id (full detail) · GET /api/hands/tags (distinct tag list)
Displayed in HandHistoryPage / AnalysisPage Color-coded pills, bar charts, drill-down by tag

4 · DB Schema & Scoping Rules

hand_tags id bigserial PK hand_id uuid FK → hands (CASCADE delete) tag text e.g. "C_BET", "OPEN_LIMP" tag_type text CHECK IN ('auto','mistake','coach','sizing') player_id uuid? FK → player_profiles (SET NULL) — null = hand-level action_id bigint? FK → hand_actions (SET NULL) — null = not per-action created_at timestamptz DEFAULT now() Unique constraints (3 partial indexes from migration 006): UNIQUE (hand_id, tag, tag_type) WHERE player_id IS NULL → hand-level UNIQUE (hand_id, tag, tag_type, player_id) WHERE player_id IS NOT NULL AND action_id IS NULL → player-level — no uniqueness constraint on action_id rows (multiple bets per hand can share a sizing tag) Indexes: idx_hand_tags_hand, idx_hand_tags_tag, idx_hand_tags_type, idx_hand_tags_hand_type, idx_hand_tags_player, idx_hand_tags_action

7 · New Capabilities to Unlock

Ideas are sorted by ease of integration within each tier. Easy = new tag in an existing analyzer or a new single-purpose analyzer (~1–3h). Medium = requires new context fields or multi-hand aggregation (~half day to 1 day). Hard = architecture change, external data, or new UI surface (> 1 day).

Easy — Drop-in Tags (new analyzer file or extend existing)

SPLIT_POT ~30 min

Fired when a hand ends in a chop (multiple winners, equal split). Useful for filtering boring hands from review queues.

hand-level StreetAnalyzer

How: Check hand.winners array length > 1 in StreetAnalyzer. No new context needed.

ALL_IN_POSTFLOP ~30 min

An all-in happened on flop, turn, or river — distinct from ALL_IN_PREFLOP. Identifies high-pressure postflop spots for review.

hand-level PostflopAnalyzer

How: Filter actions for action === 'allin' on postflop streets.

FOLD_TO_CBET ~45 min

Player folded when facing a c-bet on the flop. Reveals over-folding tendencies to continuation bets — one of the most common exploitable leaks.

player-level MistakeAnalyzer

How: Re-use findLastPFRaiser() util. Check who folded on flop to their first bet. Tag that player.

3BET_FOLD ~45 min

Player 3-bet preflop and then folded to a 4-bet. Directly measurable — coaches often look for this as a sign of bluffy 3-bet ranges.

player-level PreflopAnalyzer or MistakeAnalyzer

How: If FOUR_BET_POT is tagged, find the 3-bettor in preflop actions and check if they subsequently folded.

FLOAT_BET ~1h

Player called on the flop OOP, then led out when checked to on the turn. The "float" — calling with air then betting when opponent shows weakness.

player-level new PostflopAnalyzer extension

How: Find player who called flop OOP, then check if they bet first on turn when opponent checked. Uses positions map + byStreet.

DOUBLE_BARREL / TRIPLE_BARREL ~1h

The preflop raiser continued betting on 2 or 3 streets (flop + turn = double barrel; flop + turn + river = triple barrel). Essential for understanding aggression frequency.

hand-level PostflopAnalyzer extension

How: After C_BET detection, check if the same player also bet/raised on subsequent streets using byStreet.

STAB_BET ~1h

Bet in position when the preflop raiser checked to you on any postflop street. A "stab" at the pot when the aggressor shows weakness — fundamentally different from a c-bet or donk.

player-level new analyzer or PostflopAnalyzer

How: Player was not the PF raiser, is last to act (IP), and bets after everyone checked to them postflop.

POT_COMMITS ~1h

Player voluntarily put in > 33% of their starting stack — mathematically pot-committed. Tags the decision point where folding becomes theoretically wrong.

player + action-level new PotCommitAnalyzer

How: Track cumulative chips_in per player vs starting stack. Tag action where >33% threshold is crossed. Stack info already in context.

BLIND_STEAL ~1h

BTN, CO, or SB raised preflop and everyone folded without seeing a flop. Identifies steal attempts — coaches track steal frequency as a key metric.

player-level PreflopAnalyzer extension

How: First raise is from BTN/CO/SB, outcome is WALK (no flop). Tag the raiser. Already have positions map and WALK detection.

SIZING_CONSISTENCY ~1.5h

Player used drastically different sizing on the same street across multiple bets (e.g., pot-bet bluff, probe-bet for value). Flags exploitable size-tells.

player-level SizingAnalyzer extension

How: After sizing tags computed per action, check if the same player has 2+ bet/raise actions on same street with sizing buckets that differ by > 2 levels.

Medium — New Context Field or Aggregation Logic

RIVER_BLUFF ~2h

Player bet/raised on the river, held hole cards, and lost at showdown. The inverse of BLUFF_CATCH — identifies actual bluffs that got called off.

player + action-level HandStrengthAnalyzer extension

How: Requires hand.winners lookup at showdown + hole cards. Already available via evaluateAt(). Tag the river aggressor who lost.

CBET_FREQUENCY (per-player stat) ~3h

% of times player c-bets when they were the PF raiser and saw a flop. The most widely tracked coaching stat. Revealed over multiple hands, not per-hand.

aggregate stat new API endpoint

How: Query hand_tags — count C_BET_IP + C_BET_OOP per player / count SAW_FLOP where player was PF raiser. New DB query in PlayerRepository.

STREET_AGGRESSION_FACTOR ~3h

Per-street aggression factor: (bets + raises) / (calls + folds). The classic AF stat broken down by street — shows if a player is passive postflop.

aggregate stat AnalysisPage widget

How: Count action types per street from hand_actions. Group by player. No new tags needed — pure query stat on existing data.

LEAK_SCORE ~4h

Weighted sum of mistake tags per player session — open-limp=1pt, cold-call-3bet=2pt, drew-thin=3pt, equity-fold=3pt, etc. Single number for session quality.

aggregate stat AnalysisPage / session view

How: Define weight table in config. Query hand_tags WHERE tag_type='mistake' per session. Compute weighted sum. Surfaced as a session-level KPI.

TURN_BARREL_EQUITY ~4h

Tagged the turn barrel (double-barrel) with equity context: was the second-street bet backed by equity (>50%) or was it a pure bluff (<30%)?

player + action-level new EquityAnalyzer extension

How: Combine double-barrel detection with existing getEquityAt('turn'). Two new tags: BARREL_EQUITY_BET / BARREL_BLUFF. Needs hole cards.

POSITION_LEAK (OOP loss rate) ~5h

How often a player loses when they are OOP (SB, BB, UTG) vs IP (BTN, CO, HJ). The single most important positional leak metric.

aggregate stat new PlayerRepository query

How: Join hand_players with hand_actions.position (already stored). Compare win/loss by position group. Surfaces on player profile.

TAG_TREND (session-over-session delta) ~6h

Show whether mistake frequency is trending up or down across the last N sessions. Answers: "is this player improving?"

UI widget AnalysisPage

How: Group mistake tag counts by session, compute rolling average per player. Render as a sparkline in AnalysisPage alongside bar chart.

Hard — Architecture Change, External Data, or Major UI

RANGE_DEVIATION (GTO lite) 2–3 days

Compare actual preflop action to a bundled simplified GTO range chart. Tag hands where the player deviated significantly — wrong position open, calling too wide, 3-betting too tight.

player-level new RangeAnalyzer

How: Embed a lookup table of GTO ranges per position + action. Requires hole cards. Compare hand + position to expected range. Existing RangeParser infrastructure helps.

TILT_DETECTION 2–3 days

Detect session-level patterns that suggest tilt: increased OPEN_LIMP rate after a WHALE_POT loss, higher bet sizing variance, shorter decision sequences. A coaching red-flag system.

session-level tag new session analyzer

How: Requires session-scoped analysis pass (not per-hand). Build SessionAnalyzerService that reads last N hands per session and detects behavioral clusters.

HAND_SIMILARITY_SEARCH 3–4 days

Given any hand, find the top-5 most similar hands in history. Similarity = matching board texture tags + preflop action tags + pot type. Enables "show me more hands like this."

search feature HandHistoryPage

How: Represent each hand as a tag-vector. Use PostgreSQL array overlap queries (GIN index on tag arrays) or cosine similarity. Tags already in DB — it's a query problem.

COACH INSIGHT FEED 3–5 days

A generated per-player narrative: "Alice open-limped 4 times this session (up from 1 last week), folded to 3 c-bets, and never barreled turns." Turns tag data into plain-English coaching notes.

new CoachInsightService CoachAlertsPage

How: Define rule templates keyed on tag + threshold + trend. Render as text. Optionally pass to Claude API for prose generation. Surfaces in CoachAlertsPage.

SOLVER EV DELTA 1–2 weeks

Integrate a poker solver (e.g., PokerSolver.js, or a precomputed spot library) to estimate the EV of the actual action vs the solver's recommended action. The gold standard for coaching tools.

action-level tag new SolverAnalyzer

How: Requires an EV lookup table or embedded solver. Significant computation. Could be async/background. Would produce EV_LOSS / EV_NEUTRAL / EV_GAIN per action.
