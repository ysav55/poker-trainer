Tournament Reference

Generated 2026-04-05 · feat/phase2 branch

This document covers everything about tournaments in Poker Trainer: both tournament systems, every database table, all API endpoints, socket events, UI pages, role permissions, and known gaps. Use it to diagnose problems or plan new features.

Two-System Architecture

Poker Trainer has two distinct tournament systems that exist in parallel. They were built at different times for different purposes and do not share data.

Dimension System A — Table-Based System B — Standalone
Migration 013_tournament.sql 020_standalone_tournaments.sql
Primary tables tournament_configs, tournament_standings tournaments, tournament_players
Relationship Always tied to a tables row Independent entity — no table required
Game engine TournamentController → AutoController → GameManager (real cards dealt) Pure DB record-keeping only — no live game engine
Who creates POST /api/admin/tournaments → requires tournament:manage POST /api/tournaments → requires requireRole('coach')
Auth guard requirePermission('tournament:manage') requireRole('coach')
Socket-driven Yes — blind timer, eliminations, blind up all via Socket.io No — all updates are REST PATCH calls
Blind schedule field blind_schedule (JSONB on tournament_configs) blind_structure (JSONB on tournaments)
Standings storage tournament_standings (auto-written by TournamentController) tournament_players (manually patched by coach)
UI entry point /table/:id → TournamentInfoPanel in sidebar /admin/tournaments → TournamentSetup page
Player registration Players join the table room via socket (same as cash tables) Coach calls POST /api/tournaments/:id/register
Status lifecycle Implicit (table is open → playing → closed) Explicit: pending → running → paused → finished
Feature gate requireFeature('tournaments') None (no feature gate)
Table mode table_mode = 'tournament' N/A (not table-bound)
Player count enforced by Seats on the game table (max 9) Not enforced — unlimited registrations
Design Note

System A (table-based) is the live game system — cards are dealt, actions are taken, eliminations are automatic. System B (standalone) is closer to a tournament management board — it tracks chip counts and status without running an actual game engine. They are not connected: completing a System A tournament does not update a System B record, and vice versa.

Coach + tournament:manage inconsistency

POST /api/admin/tournaments requires tournament:manage. Whether coaches have this permission depends on the DB's role_permissions table. Per migration 008_rbac.sql, tournament:manage is granted to admin, superadmin, and referee — not coach. However, coaches do have table:manage, which unlocks the Advance Level and End Tournament controls in TournamentInfoPanel. This means: a coach can run a live tournament but cannot create one via the admin endpoint. They can create standalone tournaments via POST /api/tournaments which only requires requireRole('coach').

Database — Table-Based Tournament System

Table: tournament_configs

One row per tournament table. Created by POST /api/admin/tournaments.

CREATE TABLE tournament_configs (
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  table_id         TEXT        REFERENCES tables(id) ON DELETE CASCADE,
  blind_schedule   JSONB       NOT NULL DEFAULT '[]',
  -- e.g. [{"level":1,"sb":25,"bb":50,"ante":0,"duration_minutes":20}, ...]
  starting_stack   INT         NOT NULL DEFAULT 10000,
  rebuy_allowed    BOOLEAN     DEFAULT false,
  rebuy_level_cap  INT         DEFAULT 0,
  created_at       TIMESTAMPTZ DEFAULT now()
);
Blind Schedule Format

Each element in blind_schedule: { "level": 1, "sb": 25, "bb": 50, "ante": 0, "duration_minutes": 20 }. The level field is display-only. TournamentController iterates the array by index. duration_minutes is how long the level lasts before auto-advancing.

Table: tournament_standings

One row per (table, player). Auto-written by TournamentController._eliminatePlayer() and ._endTournament().

CREATE TABLE tournament_standings (
  id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  table_id             TEXT        REFERENCES tables(id) ON DELETE CASCADE,
  player_id            UUID        REFERENCES player_profiles(id),
  finish_position      INT,
  chips_at_elimination INT,
  eliminated_at        TIMESTAMPTZ,
  prize                NUMERIC     DEFAULT 0,  -- never written yet
  created_at           TIMESTAMPTZ DEFAULT now(),
  UNIQUE (table_id, player_id)    -- idx_tournament_standings_unique
);

Position logic: Position = (number of still-active players at time of elimination) + 1. The last survivor gets position 1 (winner). Second-to-last gets position 2, etc. Note: the prize column exists in the schema but is never written — prize calculation is not implemented.

Database — Standalone Tournament System

Table: tournaments

One row per standalone tournament. Not connected to a game table.

CREATE TABLE tournaments (
  id                  UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  name                TEXT        NOT NULL,
  status              TEXT        NOT NULL DEFAULT 'pending'
                                  CHECK (status IN ('pending','running','paused','finished')),
  blind_structure     JSONB       NOT NULL DEFAULT '[]',
  -- e.g. [{"level":1,"sb":25,"bb":50,"ante":0,"duration_minutes":20}, ...]
  current_level_index INT         NOT NULL DEFAULT 0,
  starting_stack      INT         NOT NULL DEFAULT 10000,
  rebuy_allowed       BOOLEAN     NOT NULL DEFAULT false,
  addon_allowed       BOOLEAN     NOT NULL DEFAULT false,
  created_by          UUID        REFERENCES player_profiles(id),
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  started_at          TIMESTAMPTZ,
  finished_at         TIMESTAMPTZ
);

Table: tournament_players

One row per (tournament, player). Chip counts and elimination state are updated manually via PATCH.

CREATE TABLE tournament_players (
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  tournament_id    UUID        NOT NULL REFERENCES tournaments(id) ON DELETE CASCADE,
  player_id        UUID        NOT NULL REFERENCES player_profiles(id),
  chip_count       INT         NOT NULL DEFAULT 0,
  is_eliminated    BOOLEAN     NOT NULL DEFAULT false,
  finish_position  INT,
  registered_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (tournament_id, player_id)   -- idx_tournament_players_unique
);
Standings Sort Order

getTournamentStandings() orders by: is_eliminated ASC (active players first), then chip_count DESC (richest first among active), then finish_position ASC nullsFirst: false (for eliminated players, 1st place at top).

API — Table-Scoped Endpoints

Mounted directly on the Express app. All require the tournaments feature gate.

POST /api/admin/tournaments Create a tournament table + config in one shot. Generates a tableId of the form tournament-{timestamp}, creates the tables row with mode='tournament', then creates the tournament_configs row. Auth: requireAuth + requirePermission('tournament:manage') + requireFeature('tournaments')
GET /api/tables/:id/tournament Returns { config, standings } for a tournament table. Config from tournament_configs, standings from tournament_standings joined with player_profiles. Auth: requireAuth + requireFeature('tournaments')
POST /api/tables/:id/tournament/start Looks up the tournament config, gets the in-memory TournamentController for the table, calls ctrl.start(config). This starts the blind timer and deals the first hand. Auth: requireAuth + requirePermission('tournament:manage') + requireFeature('tournaments')
POST /api/tables/:id/tournament/advance-level Calls ctrl._advanceLevel() immediately, bypassing the timer. Emits tournament:blind_up to all clients in the table room. Auth: requireAuth + requirePermission('tournament:manage') + requireFeature('tournaments')
POST /api/tables/:id/tournament/end Immediately ends the tournament. Finds the player with the highest stack among active players and declares them winner. Calls ctrl._endTournament(winnerId). Auth: requireAuth + requirePermission('tournament:manage') + requireFeature('tournaments')

API — Standalone Tournament Endpoints

All coach-mutating endpoints use [requireAuth, requireRole('coach')]. Standings read only needs requireAuth.

POST /api/tournaments Create a standalone tournament. Required body: { name, blindStructure: [...] }. Optional: startingStack (default 10000), rebuyAllowed, addonAllowed. Returns { id }. Auth: requireAuth + requireRole('coach')
GET /api/tournaments List all tournaments, newest first. Returns { tournaments: [...] }. Used by TournamentSetup to populate the list and filter by status tab. Auth: requireAuth + requireRole('coach')
GET /api/tournaments/:id Tournament detail + registered players + current level info. Returns the tournaments row, a players array joined with player_profiles, and currentLevel (the blind level at current_level_index). Auth: requireAuth + requireRole('coach')
POST /api/tournaments/:id/register Register a player. Body: { playerId }. Idempotent (returns existing row if already registered). Only works when status = 'pending'. Auth: requireAuth + requireRole('coach')
PATCH /api/tournaments/:id/status Transition tournament status. Valid values: pending, running, paused, finished. Setting running sets started_at; setting finished sets finished_at. Auth: requireAuth + requireRole('coach')
GET /api/tournaments/:id/standings Public-ish standings — any authenticated user can read. Returns { standings: [...] } from tournament_players joined with player_profiles, sorted active-first then by chip count. Auth: requireAuth only
PATCH /api/tournaments/:id/standings/:playerId Update a player's chip count, elimination status, or finish position. At least one of chipCount, isEliminated, finishPosition is required. Auth: requireAuth + requireRole('coach')
PATCH /api/tournaments/:id/level Advance the blind level index by 1 in the standalone tournament record. No-op if already at max. Returns { current_level_index, currentLevel }. Note: this only updates the DB row — it does NOT start any game timer. Auth: requireAuth + requireRole('coach')

Socket Events

All socket events are scoped to the table room (io.to(tableId).emit(...)). Only table-based tournament events use sockets. Standalone tournaments are purely REST.

Server → Client (emitted by TournamentController)

SERVER→ tournament:time_remaining Emitted when a level starts. Payload: { level: number, remainingMs: number }. Client uses this to start the countdown timer in TournamentInfoPanel.
SERVER→ tournament:blind_up Emitted when the timer expires and blinds advance. Payload: the new level object { level, sb, bb, ante, duration_minutes }. Client updates level display and restarts countdown.
SERVER→ tournament:final_level Emitted if _advanceLevel() is called but blinds are already at the last level. Payload: { level: currentLevelObject }. Client shows "FINAL LEVEL" badge.
SERVER→ tournament:elimination Emitted after each hand if a player's stack hits 0. Payload: { playerId, position, playerCount, playerName? }. Client appends to elimination feed (max 5 shown).
SERVER→ tournament:ended Emitted when ≤1 player remains OR _endTournament() is called manually. Payload: { winnerId, standings }. Client shows winner banner, then navigates to /tournament/:tableId/standings after 3 seconds.

Client → Server (socket events sent by clients)

CLIENT→ tournament:move_player Move a player between tables. Payload: { fromTableId, toTableId, playerId }. Requires coach role (checked via requireCoach(socket)). Removes player from source table's game state and seats them at the target table. Broadcasts updated game_state to both rooms. Note: no UI exists for this — it's a raw socket capability only.
CLIENT→ tournament:advance_level Fallback emit in TournamentInfoPanel if the REST advance-level call fails. Payload: { tableId }. Note: no socket handler exists for this event on the server — it is a dead emit. The correct path is the REST endpoint.
CLIENT→ tournament:end Fallback emit in TournamentInfoPanel if the REST end call fails. Note: no socket handler exists for this event on the server — also a dead emit.

TournamentController (Game Engine)

server/game/controllers/TournamentController.js extends AutoController (which extends the base controller). It is created in-memory when a tournament table is initialised and lives in SharedState.controllers (keyed by tableId).

Inheritance Chain

TournamentController
  └─ AutoController          (auto-deals next hand after a configurable delay)
       └─ BaseController     (handles join_room, socket events, table cleanup)

Key Methods

MethodTriggerWhat it does
start(config) REST POST /start Loads blind schedule, sets first level blinds on GM, starts level timer, calls gm.startGame()
_startLevelTimer() Called by start() and _advanceLevel() Emits tournament:time_remaining then sets a setTimeout for duration_minutes * 60000 ms to call _advanceLevel()
_advanceLevel() Timer fires or REST POST /advance-level Calls blindSchedule.advance(), applies new blinds to GM, emits tournament:blind_up or tournament:final_level, restarts timer
onHandComplete(handResult) After every hand ends (called by AutoController/GM) Scans for busted players, eliminates them, checks if ≤1 active → end tournament, otherwise calls super.onHandComplete() to deal next hand
_eliminatePlayer(playerId, chips) Called by onHandComplete() Writes to tournament_standings, emits tournament:elimination, calls gm.setPlayerInHand(playerId, false)
_endTournament(winnerId) Called when ≤1 player left or REST POST /end Clears level timer, records winner in tournament_standings, calls TableRepository.closeTable(), fetches final standings, emits tournament:ended
destroy() Table cleanup / idle shutdown Clears the level timer, calls super.destroy()

BlindSchedule Helper

server/game/controllers/BlindSchedule.js wraps the blind schedule array. It tracks the current level index and exposes getCurrentLevel(), advance() (returns next level or null if at end), and levelStartTime for elapsed time tracking.

Client Pages

/table/:id
The main game table page. When table_mode === 'tournament', renders TournamentInfoPanel in the right sidebar instead of the coach sidebar. The panel shows: blind level, countdown timer (with progress bar), players remaining count, elimination feed, and — for users with table:manage or tournament:manage — "Advance Level" and "End Tournament" buttons.
Guard: requires auth (via AppLayout). All roles can view; controls only show for canManage users.
/tournament/:tableId/standings
TournamentStandings.jsx — fetches GET /api/tables/:id/tournament on mount and displays final standings (finish position, player name, chips at elimination, prize if set). Automatically navigated to from TournamentInfoPanel when the tournament:ended socket event fires. Shows blind structure sheet, entrants list, and standings table.
Guard: requireAuth only (any logged-in user can view standings).
/tournament/:id/lobby
TournamentLobby.jsx — for the standalone system. Shows a single tournament's details: status badge, starting stack, blind structure sheet, entrants list with chip counts. Includes a "Start Tournament" button (PATCH status to 'running') and a countdown clock for a scheduled start time. Fetches GET /api/tournaments/:id on mount and polls GET /api/tournaments/:id/standings every 30 seconds when running.
Guard: requireAuth only for viewing. Status change requires coach role (enforced server-side).
/admin/tournaments
TournamentSetup.jsx — the coach/admin management page for standalone tournaments. Shows tournament list filterable by Upcoming/Active/Completed tabs. "New Tournament" drawer: add blind levels (auto-doubles from previous), set starting stack, rebuy/addon toggles, schedule for future start. Detail panel: player registration (manually assign by ID), status transitions, advance blind level.
Guard: SideNav only shows this link to roles with admin:access permission (admin, superadmin, coach). Route itself has no client-side guard — server enforces auth.

TournamentInfoPanel Component

Located at client/src/components/TournamentInfoPanel.jsx. Rendered inside TablePage when tableMode === 'tournament'. Gets its socket reference from useTable() context — specifically tableSocket?.socketRef (a useRef pointing to the raw socket.io instance). Does NOT take a socket prop.

Permission check used to show coach controls:

const canManage = hasPermission('table:manage') || hasPermission('tournament:manage');

End-to-End Flows

Flow A — Running a Live Tournament (Table-Based System)

1
Create the tournament table
Admin/referee calls POST /api/admin/tournaments with { name, blindSchedule, startingStack }. Server creates a tables row with mode='tournament' and a tournament_configs row. Returns { tableId, configId }.
2
Players join the table
Players navigate to /table/:tableId. Socket emits join_room. Server seats them in the game state (same as cash tables). Coach sees them in the sidebar.
3
Start the tournament
Coach/referee calls POST /api/tables/:id/tournament/start. Server loads the config, sets first-level blinds on the game manager, starts the level countdown timer, emits tournament:time_remaining to all clients, then calls gm.startGame() to deal the first hand.
4
Hands play out automatically
After each hand, TournamentController.onHandComplete() runs: detects bust-outs, records eliminations in DB, emits tournament:elimination, checks for tournament end condition, then auto-deals the next hand after a 2-second delay.
5
Blind levels advance
After duration_minutes per level, the timer fires _advanceLevel(). Server applies new blinds to GM, emits tournament:blind_up. TournamentInfoPanel updates the display and restarts the countdown. Coach can also manually advance via the "Advance Level" button (REST call).
6
Tournament ends
When only 1 player remains, _endTournament(winnerId) is called automatically. Records winner in tournament_standings, closes the table (TableRepository.closeTable()), emits tournament:ended. All clients see the winner banner and navigate to /tournament/:tableId/standings after 3 seconds.

Flow B — Managing a Standalone Tournament

1
Create the tournament
Coach goes to /admin/tournaments, clicks "New Tournament", fills in name + blind levels. Frontend calls POST /api/tournaments. Returns { id }.
2
Register players
Coach opens the tournament detail panel, enters player IDs manually, calls POST /api/tournaments/:id/register for each. Players appear in the entrants list with their starting stack.
3
Start + manage
Coach clicks Start → PATCH /api/tournaments/:id/status with { status: 'running' }. As the tournament progresses, coach manually updates chip counts via PATCH /api/tournaments/:id/standings/:playerId. Blind levels advance via PATCH /api/tournaments/:id/level.
4
Finish
Coach PATCHes last player's isEliminated = false (winner stays), sets others' finish positions, PATCHes status to finished. Players view standings at GET /api/tournaments/:id/standings.

Known Gaps & Issues

Coach cannot create live tournament tables
POST /api/admin/tournaments requires tournament:manage, which coaches don't have. Yet coaches have full in-game control (table:manage). A coach must ask an admin/referee to create the table, then the coach runs it.
Fix options: Grant tournament:manage to coach in migration, or add a dedicated requireRole('coach') path for table creation.
Dead socket fallbacks in TournamentInfoPanel
When REST calls fail, the panel emits tournament:advance_level and tournament:end via socket. No server-side socket handlers exist for these events — they silently do nothing.
Fix: Either add socket handlers for these events, or remove the fallback emits.
Prize pool is schema-only — never written
tournament_standings.prize exists (NUMERIC, default 0) but no code ever sets it. Prize calculation, distribution logic, and display on the standings page are all missing.
Two systems are disconnected
A live table-based tournament writes to tournament_standings. A standalone tournament tracks state in tournament_players. There is no way to connect them (e.g., "this standalone tournament is being run on table X"). Reports, leaderboards, and analytics cannot aggregate across both systems.
No UI for multi-table balancing
The tournament:move_player socket event exists in code and can move a player between tables preserving their chip stack. However, there is no UI to trigger it. A large tournament requiring multiple tables cannot be balanced using the current interface.
Rebuys defined in schema but not implemented
Both tournament_configs.rebuy_allowed and tournaments.rebuy_allowed exist. tournament_configs also has rebuy_level_cap. No rebuy logic exists in TournamentController — when a player busts, they are always eliminated. The UI shows a "Rebuys on" badge if set, but clicking it does nothing.
Feature gate only on table-based system
requireFeature('tournaments') is applied to /api/admin/tournaments and /api/tables/:id/tournament/*. It is not applied to /api/tournaments (standalone) routes. A school with tournaments disabled can still have standalone tournaments created by coaches.
Referees have tournament:manage but no UI access
Referees have tournament:manage (can hit the REST endpoints) but the SideNav does not show the /admin/tournaments link to them — only roles with admin:access see it. Referees would need to know the URL directly. The TournamentInfoPanel correctly shows controls to referees via the canManage check.
Standalone tournament blind timer is manual-only
In the standalone system, advancing the blind level requires a coach to manually call PATCH /api/tournaments/:id/level. There is no automatic timer — the server does not track elapsed time or auto-advance levels. For a real tournament a coach would need to watch the clock themselves.
winner_name not populated in TournamentInfoPanel
When tournament:ended fires, winner.winnerId is a UUID. The panel renders: Winner: {winner.winnerId} — displaying a raw UUID instead of the player's name. The standings fetch after the fact has the name, but the immediate banner does not.
