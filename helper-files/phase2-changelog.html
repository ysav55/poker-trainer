How It Works End-to-End

// 1. Login → JWT issued with stableId + role POST /api/auth/login → JwtService.sign({ stableId, name, role }) // 2. AuthContext fetches permissions after login GET /api/auth/permissions → ["hand:tag", "table:create", ...] // 3. Express middleware chain on guarded routes app.post('/api/playlists', requireAuth, requirePermission('playlist:manage'), handler) // 4. requirePermission fetches from DB (or cache) supabase.from('player_roles') .select('roles(role_permissions(permissions(key)))') .eq('player_id', uid) // uid = req.user.stableId // 5. Socket guards for real-time events if (!await requireSocketPermission(socket, 'hand:tag')) return;

✓ FIXED — Coach-as-Player in Non-Coached Modes

joinRoom.js now awaits the table's mode from the DB before applying coach logic. In uncoached_cash and tournament modes, isCoach is forced to false regardless of JWT role — so coaches get dealt cards and sit at the table like any player.

const tableRow = await TableRepository.getTable(tableId).catch(() => null); const mode = tableRow?.mode ?? 'coached_cash'; if (mode !== 'coached_cash') isCoach = false; // coach becomes a player

Multi-Table Strategy

One socket per table — zero server changes. Each table view opens an isolated socket.io connection via useTableSocket(tableId). Acceptable for ≤4 concurrent tables.
