Key Design Decisions

No Supabase in browserclient/src/lib/supabase.js is a Proxy stub that throws. All data goes via authenticated Express routes.
Real-time via Socket.ioGame state only. History, stats, tags travel REST.
RBAC at two layersREST: requirePermission(key). Socket: requireCoach(socket) guards.
Auto-tag pipeline10 analyzers run in parallel after every hand. Tags written atomically; coach tags never overwritten.
Stable player identitystableId UUID in localStorage. Persists across sessions and devices.
AI is optionalNarratorService degrades gracefully if ANTHROPIC_API_KEY is absent.

Backend

Auth System

Closed system — no external OAuth. Players are defined in players.csv or the DB. RBAC permissions are fetched from Supabase and cached in-memory.

Login Flow

1
POST /api/auth/login

Checks display_name + password against players.csv (primary) or the DB (fallback). Bcrypt comparison with 12 rounds.

2
JwtService.sign()

Issues 7-day JWT containing stableId, name, and role. Signed with SESSION_SECRET. Server exits at startup if key is missing.

3
Client storage

JWT stored as poker_trainer_jwt in localStorage. Injected by apiFetch() as Authorization: Bearer. stableId stored as poker_trainer_player_id.

4
Middleware enforcement

REST: requireAuth verifies JWT → requirePermission('key') checks DB cache. Socket: socketAuthMiddleware (io.use()) + requireCoach(socket) per-event guards.

No Supabase in the browser. client/src/lib/supabase.js is a Proxy stub that throws on access. All DB reads go through authenticated Express routes.

Dual Auth Paths

ISS-99: Two auth paths currently run simultaneously. players.csv is primary (bcrypt lookup). The DB-backed migration exists but is not fully cut over. Both work in parallel.

players.csv (primary)

Format
name,bcrypt_hash,role alice,$2b$12$...,coach bob,$2b$12$...,player

After editing, run: node scripts/hash-passwords.js

DB-backed (secondary)

Self-registered users (via /register) only exist in the DB. coaches registered via /register-coach also land in the DB. CSV users do not have DB rows unless manually added.

Permission Cache

On first authed request, getPlayerPermissions(playerId, roleHint) fetches from player_roles → roles → role_permissions → permissions and caches the result in a server-side Map. Subsequent requests hit the cache.

Cache is invalidated via invalidatePermissionCache(playerId) — called automatically after role changes via admin endpoints.

Coach Intelligence Services

Five services power the coaching intelligence layer. They sit between the API routes and the DB, consuming baseline stats to generate alerts, reports, and briefs. The AI narrator is optional.

BaselineService
server/services/BaselineService.js
Computes and stores 30-day rolling baselines for every student. Foundation for all alert and report accuracy.
  • Writes student_baselines table
  • Metrics: VPIP, PFR, 3-bet %, AF, mistake rates, hands/week
  • Required before alerts or reports yield meaningful data
AlertService
server/services/AlertService.js
Detects coaching-relevant events by comparing current metrics against baselines. Returns severity-ranked alert list.
  • InactivityDetector — No recent sessions
  • VolumeDropDetector — Hands/week declined
  • MistakeSpikeDetector — Mistake rate increased
  • LosingStreakDetector — Recent sessions losing
  • RegressionDetector — Key stat regression
  • MilestoneDetector — Positive achievements
  • Ignores alerts below 0.2 severity (milestones bypass)
  • Deduplicates — updates existing rather than inserting
ProgressReportService
server/services/ProgressReportService.js
Generates weekly, monthly, or custom date-range progress reports with 8 sections and a 0–100 grade.
  • Core stats: VPIP, PFR, 3-bet %, WTSD, W$SD, aggression
  • Mistake rates: open limp %, cold call 3-bet %, equity fold %
  • Leak evolution (improvement / regression delta)
  • Idempotent upsert into progress_reports
  • Passes data to NarratorService for narrative summary
  • Stable overview aggregates across all students
SessionPrepService
server/services/SessionPrepService.js
Generates a pre-session brief for an upcoming coaching session. Cached and refreshable.
  • Recent leaks (last 50 hands)
  • Rolling 30-day baseline stats
  • Active alerts for the student
  • Session streak (hours since last play)
  • Total + weekly hand counts
  • Passes to NarratorService for readable paragraph
NarratorService
server/ai/NarratorService.js
Turns structured coaching data into readable prose using Claude Haiku. Never in a critical path — always optional.
  • Model: claude-haiku-4-5-20251001
  • narrateAlerts() — 3–5 sentence alert digest
  • narratePrepBrief() — session overview paragraph
  • narrateProgressReport() — report summary
  • narrateStableOverview() — stable-wide summary
  • Returns null if API key absent or call fails

Hand Analyzer

10 independent analyzers run in parallel after every hand ends. Tags are written atomically to hand_tags. Coach tags are never touched by the auto-analyzer.
