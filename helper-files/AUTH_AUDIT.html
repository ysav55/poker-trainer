🔐 Auth System Audit

Poker Trainer · feat/phase2 · Full code audit as of 2026-04-05

Auth Model

JWT-based, server-issued, 7-day expiry. HS256 signed with SESSION_SECRET.

Access Control

Two-layer: requireRole() for coarse role checks, requirePermission() for 12 fine-grained keys.

Identity

Stable UUID in DB (player_profiles.id). Stored in localStorage on client.

🗺 System Overview

The auth system is fully custom — no Supabase Auth or OAuth. Every token is issued by the Express server and verified server-side. The browser never touches the Supabase service-role key.

Key design decisions

  • Single JWT secret: SESSION_SECRET env var. Server exits on startup if missing.
  • No refresh tokens: 7-day sliding expiry. Expired tokens force re-login.
  • No anon access to Supabase: Client uses apiFetch() → Express → Supabase service-role key. A Proxy stub throws on any direct Supabase import in the browser.
  • Dual identity: Socket connections use socket.id internally, but stableId (UUID from JWT/localStorage) is what gets written to the DB.
  • Spectators: Unauthenticated users can connect via Socket.IO, but only as spectators. They cannot act at the table.
  • Bot role: Server-spawned bot sockets bypass human-join checks. Role 'bot' is set in JWT.

🔑 JWT Service — server/auth/JwtService.js

Token structure (payload)

{
  stableId: "uuid",   // player_profiles.id
  name:     "Alice",  // display_name
  role:     "coach"   // primary role name
}

Plus standard JWT claims: iat, exp.

Configuration

FieldValue
AlgorithmHS256 (jsonwebtoken default)
Expiry7 days
Secret sourceprocess.env.SESSION_SECRET
Required?FATAL if missing — server exits
Full source — JwtService.js 
'use strict';
const jwt = require('jsonwebtoken');
const SECRET = process.env.SESSION_SECRET;

function sign(payload) {
  return jwt.sign(payload, SECRET, { expiresIn: '7d' });
}

function verify(token) {
  if (!token) return null;
  try {
    return jwt.verify(token, SECRET);
  } catch {
    return null;   // expired, tampered, or missing
  }
}

module.exports = { sign, verify };
⚠ No token revocation. JWTs are stateless — a stolen token is valid until it expires (7 days). There is no logout endpoint, no token blacklist. Logging out on the client just deletes localStorage.poker_trainer_jwt.

🛡 HTTP Middleware

requireAuth — server/auth/requireAuth.js

The fundamental gate. Reads Authorization: Bearer <token>, verifies via JwtService, and attaches the decoded payload to req.user.

What it sets on req.user

{
  stableId: "uuid",
  name:     "Alice",
  role:     "coach",
  iat:      1234567890,
  exp:      1235172690
}

Failure responses

ConditionResponse
No / malformed header401 auth_required
Invalid / expired token401 invalid_token
Full source — requireAuth.js 
'use strict';
const JwtService = require('./JwtService');

function requireAuth(req, res, next) {
  const auth = req.headers.authorization;
  if (!auth || !auth.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'auth_required', message: 'Login required' });
  }
  const payload = JwtService.verify(auth.slice(7));
  if (!payload) {
    return res.status(401).json({ error: 'invalid_token', message: 'Session expired — please log in again' });
  }
  req.user = payload;
  next();
}

module.exports = requireAuth;

requireRole — server/auth/requireRole.js

Middleware factory for coarse role-level gating. Must follow requireAuth. Does a strict equality check on req.user.role.

⚠ Exact match only. requireRole('coach') will reject admin and superadmin. There is no role hierarchy in this middleware — only the permission system in requirePermission handles that.

Usage in routes

// coach-only alert endpoints
app.get('/api/coach/alerts', requireAuth, requireRole('coach'), handler);

// coach-only playlist mutations
app.post('/api/blind-presets', requireAuth, requireRole('coach'), handler);

// coach-only announcements
app.post('/api/announcements', requireAuth, requireRole('coach'), handler);
Full source — requireRole.js 
'use strict';

function requireRole(role) {
  return function (req, res, next) {
    if (!req.user || req.user.role !== role) {
      return res.status(403).json({ error: 'forbidden', message: `Requires role: ${role}` });
    }
    next();
  };
}

module.exports = requireRole;

requirePermission — server/auth/requirePermission.js

Fine-grained permission middleware backed by the DB. Queries player_roles → roles → role_permissions → permissions. Results are cached in-memory per player.

Cache behaviour

  • In-memory Map<playerId, Set<permKey>>
  • Populated on first permission check per player
  • Invalidated explicitly via invalidatePermissionCache(playerId)
  • Called after any role change in admin/users.js
  • No TTL — cache lives until server restart or explicit invalidation

CSV fallback

If the DB query returns zero permission keys (e.g. player created via old CSV path with no player_roles rows), it falls back to a role-name lookup on the roles table using the JWT's role field as a hint.

Full source — requirePermission.js 
'use strict';
const supabase = require('../db/supabase.js');

// In-memory cache: Map<playerId, Set<permKey>>
const permissionCache = new Map();

async function getPlayerPermissions(playerId, roleHint = null) {
  if (permissionCache.has(playerId)) return permissionCache.get(playerId);

  const { data } = await supabase
    .from('player_roles')
    .select('roles(role_permissions(permissions(key)))')
    .eq('player_id', playerId);

  let keys = new Set(
    data?.flatMap(pr =>
      pr.roles?.role_permissions?.map(rp => rp.permissions?.key).filter(Boolean) ?? []
    ) ?? []
  );

  // Fallback: players with no player_roles rows → derive from JWT role name
  if (keys.size === 0 && roleHint) {
    const { data: roleData } = await supabase
      .from('roles')
      .select('role_permissions(permissions(key))')
      .eq('name', roleHint)
      .single();
    keys = new Set(
      roleData?.role_permissions?.map(rp => rp.permissions?.key).filter(Boolean) ?? []
    );
  }

  permissionCache.set(playerId, keys);
  return keys;
}

function invalidatePermissionCache(playerId) {
  permissionCache.delete(playerId);
}

function requirePermission(...keys) {
  return async (req, res, next) => {
    const uid = req.user?.stableId ?? req.user?.id;
    if (!uid) return res.status(401).json({ error: 'Unauthorized' });
    const perms = await getPlayerPermissions(uid, req.user?.role);
    if (keys.every(k => perms.has(k))) return next();
    return res.status(403).json({ error: 'Insufficient permissions' });
  };
}

module.exports = { getPlayerPermissions, invalidatePermissionCache, requirePermission };

Full source — socketAuthMiddleware.js 
'use strict';
const JwtService = require('./JwtService');

function socketAuthMiddleware(socket, next) {
  const token = socket.handshake.auth?.token || '';
  const payload = JwtService.verify(token);

  if (payload) {
    socket.data.authenticated = true;
    socket.data.stableId      = payload.stableId;
    socket.data.role          = payload.role;
    socket.data.isCoach       = payload.role === 'coach';
    socket.data.isBot         = payload.role === 'bot';
    socket.data.jwtName       = payload.name;
  } else {
    socket.data.authenticated = false;
  }

  next();  // always call next() — spectators allowed through
}

module.exports = socketAuthMiddleware;

socketGuards — server/auth/socketGuards.js

Guards for individual socket event handlers. requireCoach() is used in 30+ places across handler files.

requireCoach(socket, action)

Checks socket.data.isCoach. If not a coach, emits an error event and returns true (caller should return). Returns false if allowed.

// Usage pattern in all handler files:
if (requireCoach(socket, 'start the game')) return;

requireSocketPermission(socket, ...keys)

Async variant — queries the DB permission cache for the socket's playerId. Returns false and emits error if lacking permissions.

// Note: uses socket.data.playerId (not stableId)
// This is a different field name than HTTP middleware!
const ok = await requireSocketPermission(socket, 'hand:tag');
if (!ok) return;
⚠ Field name inconsistency. socketPermissions.js reads socket.data.playerId, but socketAuthMiddleware.js sets socket.data.stableId. These are different field names. If requireSocketPermission is called on a newly-connected socket (before join_room sets socket.data.playerId), it will always fail with "Not authenticated".

The authentication entry point. Validates credentials and returns the player identity for JWT signing.

Full source — PlayerRoster.js 
'use strict';

const { findByDisplayName, getPrimaryRole } = require('../db/repositories/PlayerRepository');
const bcrypt = require('bcrypt');

async function authenticate(name, password) {
  if (!name || !password) return null;

  const player = await findByDisplayName(name.trim());
  if (!player || !player.password_hash) return null;

  // Reject suspended or archived accounts
  if (player.status === 'suspended' || player.status === 'archived') return null;

  const valid = await bcrypt.compare(password, player.password_hash);
  if (!valid) return null;

  const role = await getPrimaryRole(player.id);
  return { id: player.id, name: player.display_name, role: role ?? 'player' };
}

function load()   { /* no-op */ }
function reload() { /* no-op */ }
function getRole(_name) { return null; }

module.exports = { authenticate, load, reload, getRole };

⏳ Trial System

Self-registered students start on a trial. Limits are set at registration time and enforced at the join_room socket handler (not currently shown in joinRoom.js — enforcement may be in a separate location or feature-flagged).

Trial limits

  • Duration: 7 days (TRIAL_DAYS = 7)
  • Hands: 20 hands (TRIAL_HANDS = 20)

DB fields (player_profiles)

  • trial_expires_at — TIMESTAMPTZ
  • trial_hands_remaining — INTEGER

Trial role path

Registration with coachId → role = coached_student

Registration without coachId → role = solo_student

Both are trial accounts. Role trial is not assigned by the registration route — it's available for admin use.

💻 Client Side

apiFetch — client/src/lib/api.js

All API calls go through this single function. It reads the JWT from localStorage and attaches it as a Bearer token.

Token storage

  • JWT stored in localStorage.poker_trainer_jwt
  • Player UUID in localStorage.poker_trainer_player_id
  • On 401 with token present → clears both + throws "Session expired"
  • On 401 without token → throws HTTP error (login attempt failure)

Environment config

const API_BASE = import.meta.env.DEV
  ? 'http://localhost:3001'
  : '';  // same origin in production

In production, Express serves the built React app. No CORS issues.

Full source — api.js 
const API_BASE = import.meta.env.DEV ? 'http://localhost:3001' : '';

export async function apiFetch(path, options = {}) {
  const token = localStorage.getItem('poker_trainer_jwt');
  const res = await fetch(`${API_BASE}${path}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...options.headers,
    },
  });
  if (!res.ok) {
    let message = `HTTP ${res.status}`;
    try { message = (await res.json()).message || message; } catch { /* not JSON */ }
    if (res.status === 401) {
      const hadToken = !!localStorage.getItem('poker_trainer_jwt');
      localStorage.removeItem('poker_trainer_jwt');
      localStorage.removeItem('poker_trainer_player_id');
      const err = new Error(hadToken ? 'Session expired — please log in again' : message);
      err.status = 401;
      throw err;
    }
    throw new Error(message);
  }
  return res.json();
}

Socket connection

The socket sends the JWT at connect time via socket.handshake.auth.token:

// In useConnectionManager.js (inferred from architecture)
const socket = io(SERVER_URL, {
  auth: { token: localStorage.getItem('poker_trainer_jwt') }
});
Token refresh gap: If the JWT expires while the user has a socket connection open, the socket remains connected (no re-auth on expiry). The token is only re-validated on new HTTP requests via apiFetch. A new socket connection would fail after the 7-day expiry.

Supabase Row Level Security is configured, but since the Express server uses the service-role key, RLS is bypassed for all server-side operations. RLS only applies if a client were to connect directly to Supabase — which is prevented by the Proxy stub in the client.

The policies that exist:

  • player_profiles: authenticated read all, own update
  • hands, hand_actions, hand_tags: authenticated read
  • hand_players: coaches see all, students see own rows
  • sessions, leaderboard, playlists: authenticated read

🔍 Security Notes & Observations

Strengths

  • ✅ Single JWT module: Only JwtService.js imports jsonwebtoken. No accidental key usage elsewhere.
  • ✅ Server exits if SESSION_SECRET missing: No silent degradation to an empty secret.
  • ✅ bcrypt rounds=12: Strong hashing for all passwords.
  • ✅ Rate limiting on auth endpoints: 20 requests per 15 minutes.
  • ✅ No Supabase keys in browser: Client Proxy stub throws on any Supabase import. All DB access goes through Express.
  • ✅ Soft deletes only: Archived accounts keep their data. No cascading deletes on auth operations.
  • ✅ Permission cache invalidation: Called after every role change to prevent stale permission grants.
  • ✅ Duplicate tab detection: Old socket kicked when same stableId joins from a second window.
  • ✅ Password confirm on destructive operations: /api/auth/deactivate requires current password re-entry.

Potential issues & gaps

  • ⚠ No token revocation: Stolen tokens are valid for 7 days. No logout endpoint, no blacklist. If an admin suspends an account, the suspended user can still make HTTP requests with their existing JWT until it expires.
  • ⚠ Admin password reset — no current password: POST /api/admin/users/:id/reset-password lets any user:manage admin reset anyone's password without knowing the old one. This is expected for admin workflows but worth noting.
  • ⚠ requireRole() is exact match only: requireRole('coach') blocks admin/superadmin. This means superadmin can't access coach-only endpoints without a role change. Check if this is intentional.
  • ⚠ socketPermissions.js reads socket.data.playerId: The field set by socketAuthMiddleware is socket.data.stableId. Calling requireSocketPermission before join_room populates socket.data.playerId will silently fail as "Not authenticated".
  • ⚠ Pending coach accounts can log in: The authenticate() check only blocks 'suspended' and 'archived'. A pending coach application account can log in and receive a JWT. Their permissions will be 0 (no role assigned), but they can still hit requireAuth-only endpoints.
  • ⚠ No CSRF protection: JWT in Authorization header (not cookies) means CSRF is not a concern for API calls. But the deactivate endpoint uses body parameters — if a route were accidentally cookie-auth'd, CSRF would apply. Not currently an issue.
  • ⚠ Permission cache has no TTL: Cache lives until server restart or explicit invalidatePermissionCache() call. Multi-instance deployments (Fly.io scaling to multiple machines) will have stale caches across instances.
  • ⚠ CORS origin from env: If CORS_ORIGIN is not set in production, all cross-origin requests are blocked (which is safe), but there's a warning log — not a hard startup failure like SESSION_SECRET.
  • ⚠ trial_hands_remaining enforcement path unclear: The DB fields exist and are set at registration, but the active enforcement code (checking/decrementing hands during play) should be verified.
