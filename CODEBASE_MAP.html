1

The 7 Storage Buckets

Everything the app remembers lives in one of these seven buckets. Think of each as a filing cabinet drawer with a specific purpose.

🧑
Player Accounts
  • Display name chosen at registration
  • Email address used to log in
  • Password stored as a secure one-way hash — never the real text
  • A permanent unique ID that travels with the player forever
  • Timestamp of the last time this player logged in
🎮
Sessions
  • Created once each time the server starts up
  • Records which table (room) the session belongs to
  • Captures the exact moment the session began
  • All hands played during that session link back here
🃏
Hands
  • One record per completed poker hand
  • Start and end timestamps
  • All five community cards (the “board”)
  • Final pot size and who won
  • Tags — both auto-detected patterns and coach notes
👥
Players in Hand
  • One record per player, per hand
  • Chip count at the start and end of the hand
  • The two hole cards they were dealt
  • Whether they won the pot
  • VPIP and PFR flags (did they invest? did they raise pre-flop?)
⚡
Actions
  • Every single move: fold, call, check, raise, or all-in
  • Which street it happened on (pre-flop, flop, turn, river)
  • The chip amount involved
  • A “reverted” flag if the coach later undid the action
  • Exact timestamp of each move
📋
Playlists
  • A named collection the coach creates for review sessions
  • Stores the playlist's display name
  • Tracks which session it was created in
  • Can be activated or deactivated for study mode
🔗
Playlist Entries
  • The glue between Playlists and Hands
  • Records which hand belongs to which playlist
  • Stores the position so the coach can reorder hands
  • Allows one hand to appear in multiple playlists

The User Journey — What Gets Saved & When

Follow a player from first registration through to reviewing hands in a playlist. Each step shows exactly what the system records.

1
🔐
Player Registers
  • Name, email, and a hashed password written to Player Accounts
  • A permanent unique ID is generated and returned to the browser, saved in local storage
  • This ID follows the player across all future visits — even across devices when they log in
Session begins
2
🔑
Player Logs In
  • Name and password are checked against Player Accounts
  • The “last seen” timestamp is updated in Player Accounts
  • The permanent ID is returned to the browser and stored for the session
Connects to the table
3
🪑
Player Joins the Table
  • Server confirms the account exists in Player Accounts
  • “Last seen” refreshed again — the player is online
  • The browser ID is linked to the current connection so actions can be attributed correctly
First hand about to begin
4
🃏
A Hand Starts
  • A new record is created in Hands, stamped with start time and linked to the current Session
  • One record per seated player is created in Players in Hand, recording their chip count at this moment
  • Hole cards are not saved yet — they are recorded when the hand concludes
Betting rounds begin
5
⚡
Players Act
  • Every fold, call, check, and raise is written to Actions the instant it happens
  • Each record includes the street name, chip amount, and a precise timestamp
  • Nothing is held in memory and flushed later — each action is permanent as soon as it is taken
Coach steps in
6
↩️
Coach Undoes an Action
  • The undone action in Actions is flagged as “reverted” — it is never deleted
  • This preserves a full audit trail of what actually happened at the table
  • Later, the system detects undo usage and adds an UNDO_USED tag to the hand
Hand reaches showdown or everyone folds
7
🏆
Hand Ends
  • The Hands record is updated with end time, all board cards, final pot size, and the winner
  • Each player's record in Players in Hand is updated with ending chip count, hole cards, win flag, and VPIP/PFR flags
  • WTSD/WSD flags (went to showdown / won at showdown) are now saved correctly — computed from showdown_result.allHands[] membership and persisted to hand_players
System scans the completed hand
8
🏷️
Auto-Tags Applied
  • The system scans every saved action and looks for known patterns: WALK, 3BET_POT, C-BET, CHECK_RAISE, BLUFF_CATCH, WHALE_POT
  • If the coach used undo at any point, an UNDO_USED tag is also added
  • All detected tags are written to the Hands record automatically — no coach input needed
Coach reviews the hand
9
🗒️
Coach Tags the Hand
  • Coach clicks tag buttons in the UI to annotate the hand with their own labels
  • These tags are merged with the auto-detected ones and saved to the Hands record
  • Tags are the primary way to filter and search through hand history later
Coach organises hands for review
10
📋
Hand Added to a Playlist
  • Coach creates or selects a Playlist (e.g. “River mistakes — week 3”)
  • A Playlist Entry is created, linking this hand to the playlist at a specific position
  • The same hand can appear in multiple playlists — each gets its own entry record

How the Buckets Connect

Each arrow shows a “parent → child” relationship. One parent record can link to many child records.

* Convention-only links: Actions and Players-in-Hand records reference Player Accounts by their unique ID, but the database does not currently enforce this link. If an account were deleted, those records would have no matching player. This is a known gap being addressed — see Section 4, Gap 1.
4

Known Gaps — All Resolved

All data-layer gaps identified in the initial audit were fixed by 2026-03-16. The table below shows what each gap was and how it was addressed.

Gap 1 ✓
Missing Referential Integrity
Still convention-based (no FK enforcement at SQLite level), but a case-insensitive UNIQUE index on display_name was added (DB-03). Full FK enforcement remains a future consideration if the app moves to a larger DB.
Gap 2 ✓
VPIP & PFR from Wrong Moment — Fixed
Fixed 2026-03-15 (GAP-2). VPIP/PFR now read exclusively from preflop action records, not the player’s last action of the hand. Career stats are accurate.
Gap 3 ✓
WTSD & WSD Now Persisted
Fixed 2026-03-15 (GAP-3) and refined 2026-03-16 (DB-01). WTSD/WSD computed from showdown_result.allHands[] membership (not the is_active flag) and written to hand_players on every hand end.
Gap 4 ✓
UNDO_USED Tag Now Returned
Fixed 2026-03-15 (GAP-4). The tag query now includes mistake_tags in its response — UNDO_USED is visible in the Stats Dashboard and session reports.
Gap 5 ✓
Fast Lookup Index Added
Fixed 2026-03-15 (GAP-5). An index on hand_players.player_id was added to Database.js. Player history queries are now O(log n) instead of full-table scans.
Gap 6 ✓
WHALE_POT Uses Real Big Blind
Fixed 2026-03-15 (DB-02 + GAP-7). The big_blind value is now correctly passed from index.js to HandLogger.startHand() and used in WHALE_POT threshold calculations.
